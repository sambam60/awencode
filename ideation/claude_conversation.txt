Sam, this is a really interesting design problem — and you're right that nobody has nailed it yet. The current paradigm is basically "chat list with extra steps," which completely fails to capture the actual workflow you're describing.
Let me think through what you're actually articulating, because I think the core insight is sharper than just "kanban for agents."
The real structure you're describing isn't a chat list — it's a workday with hierarchy.
You have one orchestrator context that represents your intent for the day, and then N worker contexts that each own a discrete unit of work. The orchestrator needs to know the state of all workers, and workers optionally need awareness of siblings. That's not a sidebar. That's a tree with shared state.
The reason every existing tool feels wrong is they all treat each agent session as a peer — flat list, maybe with pins or tabs. But your mental model is fundamentally hierarchical: there's a conductor and there are players. The UX should reflect that.
A few design directions worth exploring:
The orchestrator as a persistent dashboard, not a chat. Your "main chat" isn't really a chat — it's a command center. It should show the state of all active work streams at a glance (queued, in progress, in review, deployed), let you send instructions to sub-agents, and surface when something needs your attention. The chat interface is just one interaction mode within it, not the whole thing.
Sub-agents as cards, not conversations. Each feature/issue agent should be a stateful card that the orchestrator can see: current status, last action, branch name, whether it's blocked. You expand into the full conversation when you need to, but the default view is the state, not the transcript. This is where the kanban instinct is right — but it's kanban of agent states, not tasks.
Shared context layer. Your point about the nth run starting from scratch is the biggest efficiency kill. There should be a "workspace context" that persists across all agents for a session/day: repo structure, what's been deployed, what's in flight, conventions established. Each agent reads from it by default, writes back summaries. The orchestrator curates it. This is basically the missing primitive — a shared memory space that sits above individual agent conversations.
Auto-lifecycle management. When a sub-agent's PR is merged and deployed, it shouldn't need you to unpin it. The orchestrator should detect the state change and archive it, maybe with a "deployed today" log. Your pin workflow is you manually maintaining state that the system should track.
The hard design question is the orchestrator interaction model. Is it a chat you talk to that happens to have a dashboard? Is it a dashboard with a chat embedded? I'd lean toward the latter — the primary view is the state of your workday, and you drop into conversational mode when you need to give complex instructions or reason through something. The MercuryOS-style intent-driven interface you explored before is actually relevant here: the orchestrator should infer what you need based on the current state rather than waiting for explicit commands.

Now map this directly onto your agent orchestrator problem. Every coding agent GUI right now — Codex, Cursor, Conductor, OpenCode — is a conversational interface pretending to be a workflow tool. You type what you want in natural language, the agent responds in natural language, and your entire orchestration layer is... scrolling through chat history. That's exactly the anti-pattern Julian is describing.
Here's what this reframes for your design:
The orchestrator shouldn't be a chat. It should be a dashboard where chat is one compressed input mode. Your "main chat" workflow is fighting the medium. You don't want to converse with an orchestrator — you want to glance at a board and know what's happening, drag a card from "review" to "deploy," and maybe type a short command when something needs nuance. The conversational interface should only appear when you're doing genuine thinking work with the agent — the equivalent of Julian walking around talking to ChatGPT voice mode to develop ideas. For status tracking, task assignment, and lifecycle management, you want the equivalent of keyboard shortcuts and glanceable state — not a chat thread.
The sub-agent threads are where conversation makes sense, but even there it should be compressed. When you open a feature thread, you don't need to re-read 40 messages of back-and-forth. You need: current status, last meaningful action, branch/PR link, blockers. The conversation is the implementation detail behind the card. You expand into it when you need to, like expanding a collapsed section.
The "nth run amnesia" problem is literally Julian's butter-passing metaphor. After 50 years of breakfast, the grandfather doesn't ask what his wife wants — he just knows. Your orchestrator should accumulate workspace context throughout the day and pass it implicitly. Not through conversation, but through a shared context layer that every agent reads from automatically. The agent shouldn't need to "introduce itself to the repo" on run 5 — it should already know the repo, the conventions, what's deployed, what's in flight.
The pin/unpin workflow is you manually maintaining state that should be automatic. Julian's whole point about speed and convenience always winning — you're spending cognitive cycles on bookkeeping that the system should handle. Deploy detection → auto-archive. PR merged → card moves. No conversation needed.
So the design shape I'd think about: a spatial workspace (not a chat list) where agent tasks live as stateful cards on a board. The board itself is the orchestrator view — columns for queued, active, review, deployed. Each card shows compressed state. You can issue commands to cards (approve, deploy, pause, reassign) through direct manipulation or short keyboard commands, not by opening a chat and typing a paragraph. The chat interface exists within each card for when you need the full conversational depth, and as a global command bar (like Raycast/Linear's Cmd+K) for complex instructions that don't map to a button.
